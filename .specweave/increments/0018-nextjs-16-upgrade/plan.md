# Implementation Plan: Upgrade Next.js to v16

## Overview

Upgrade `next` from `^15.1.0` to `^16.1.6` in both `apps/web` and `apps/admin`. The sole breaking change affecting this codebase is the removal of `transpilePackages` config option. Both internal packages (`@shopvui/ui`, `@shopvui/shared`) already ship pre-built via tsup with proper `exports` fields, so Next.js 16's native package resolution handles them without any config -- `transpilePackages` lines are simply deleted.

No new APIs, no routing changes, no React version bump required (React 19 is already in use and compatible with Next.js 16).

## Architecture

This is a configuration-only change. No component boundaries, data models, or APIs are affected.

### Affected Files

| File | Change |
|------|--------|
| `apps/web/package.json` | `next` version `^15.1.0` -> `^16.1.6` |
| `apps/admin/package.json` | `next` version `^15.1.0` -> `^16.1.6` |
| `apps/web/next.config.ts` | Remove `transpilePackages` line |
| `apps/admin/next.config.ts` | Remove `transpilePackages` line |
| `pnpm-lock.yaml` | Regenerated by `pnpm install` |

### Why transpilePackages Removal Is Safe

Both `@shopvui/ui` and `@shopvui/shared` already:
1. Build to `dist/` via tsup (pre-compiled JS, not raw TS)
2. Declare proper `exports` maps with `types`, `import`, and `require` conditions
3. Set `"type": "module"` for ESM-first resolution

Next.js 16 resolves workspace packages through their `exports` field. Since the packages already export compiled output (not source TS), no transpilation by Next.js is needed. The `transpilePackages` config was technically redundant even on v15 given the pre-built setup, but removing it is now mandatory since v16 drops the option.

## Technology Stack

- **Framework**: Next.js 16.1.6
- **Package manager**: pnpm (workspace protocol, lockfile regeneration)
- **Build orchestration**: Turborepo (cache must be cleared after upgrade)
- **React**: 19.x (no change needed)

## Implementation Phases

### Phase 1: Version Bump and Config Update
1. Update `next` version in both `apps/web/package.json` and `apps/admin/package.json`
2. Remove `transpilePackages` from both `next.config.ts` files
3. Run `pnpm install` to regenerate lockfile
4. Clear turborepo cache: `npx turbo clean` or `rm -rf node_modules/.cache/turbo`

### Phase 2: Build Verification
1. Run `turbo build` -- confirm both apps compile successfully
2. Run `next dev` for each app -- confirm dev server starts without errors
3. Verify PWA service worker generation (`@serwist/next`) in production build

### Phase 3: Third-Party Compatibility Verification
1. Verify `next-intl` locale routing in `apps/web`
2. Verify `next-themes` theme switching in `apps/web`
3. Verify `@tanstack/react-query` data fetching in `apps/web`
4. Verify `recharts` chart rendering in `apps/admin`

## Testing Strategy

This is a dependency upgrade, not a feature build. Testing is verification-focused:

- **Build gate**: `turbo build` must succeed for both apps (covers AC-US3-01)
- **Dev smoke test**: `next dev` starts cleanly for both apps (covers AC-US3-02)
- **Existing unit tests**: `pnpm -r run test` must pass (regression check)
- **Manual verification**: Spot-check third-party integrations in dev mode (US-004 ACs)
- **PWA check**: Confirm `sw.js` is generated in production build output (AC-US3-03)

No new tests are needed. Existing test suites serve as the regression safety net.

## Technical Challenges

### Challenge 1: Peer Dependency Conflicts
**Risk**: `@serwist/next` or `next-intl` may declare `next` peer dependency as `^15.x`, causing pnpm install warnings or errors.
**Mitigation**: Check each package's peer dependency range. If pinned to v15, check for updated versions that support v16. `@serwist/next@^9.5.6` and `next-intl@^4.8.3` are recent enough that v16 support is likely. If not, use `pnpm.peerDependencyRules.allowedVersions` in the root `package.json` as a temporary override.

### Challenge 2: Serwist Webpack Plugin Compatibility
**Risk**: If Next.js 16 changes its webpack configuration API, the `@serwist/next` wrapper may break.
**Mitigation**: Test production build first. If Serwist fails, check for a newer Serwist version or temporarily disable PWA in the build while waiting for upstream fix.

## Decisions

- **No ADR needed**: This is a straightforward major version bump with a single, well-understood breaking change. No architectural decisions are being made.
- **No domain skill delegation needed**: No frontend component work, no backend changes. Pure config/dependency update.
