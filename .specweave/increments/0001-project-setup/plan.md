# Implementation Plan: ShopVui Monorepo Scaffold

## Overview

This plan defines the architecture for a Turborepo + pnpm monorepo hosting three applications (web storefront, admin panel, NestJS API) and two shared packages (types/utils, UI components). The scope is strictly scaffolding: runnable skeletons with zero business logic, shared tooling configs, and a Docker Compose setup for local development. No database, no auth, no domain features.

## Architecture

### Repository Structure

```
shopvui/
├── apps/
│   ├── web/              # Next.js 15 App Router — storefront PWA
│   ├── admin/            # Next.js 15 App Router — admin panel
│   └── api/              # NestJS — REST API with Swagger
├── packages/
│   ├── shared/           # TypeScript types + utility functions
│   └── ui/               # React component library (shared across web & admin)
├── tooling/
│   ├── eslint/           # Shared ESLint config package
│   ├── typescript/       # Shared tsconfig base
│   └── prettier/         # Shared Prettier config package
├── turbo.json            # Pipeline definitions
├── pnpm-workspace.yaml   # Workspace globs
├── docker-compose.yml    # Local dev stack
├── package.json          # Root scripts + devDependencies
└── .prettierrc           # Points to tooling/prettier config
```

### Component Boundaries

| Component | Type | Depends On | Exposes |
|-----------|------|------------|---------|
| `apps/web` | Next.js 15 App | `packages/shared`, `packages/ui` | Storefront pages, PWA manifest |
| `apps/admin` | Next.js 15 App | `packages/shared`, `packages/ui` | Admin dashboard pages |
| `apps/api` | NestJS App | `packages/shared` | REST endpoints, Swagger at `/api/docs` |
| `packages/shared` | Library | None (leaf) | TypeScript types, utility functions |
| `packages/ui` | React Library | `packages/shared` | React components |
| `tooling/eslint` | Config Package | None (leaf) | ESLint flat config |
| `tooling/typescript` | Config Package | None (leaf) | Base `tsconfig.json` |
| `tooling/prettier` | Config Package | None (leaf) | Prettier config |

### Dependency Graph

```
tooling/eslint ─────────────────────────────┐
tooling/typescript ─────────────────────────┐│
tooling/prettier ──────────────────────────┐││
                                           │││
packages/shared (leaf)  ◄──────────┐       │││
    │                              │       │││
    ▼                              │       │││
packages/ui ────────────┐          │       │││
    │                   │          │       │││

apps/web            apps/admin   apps/api
```

Turborepo respects this DAG: `packages/shared` builds first, then `packages/ui`, then all three apps in parallel.

### Data Model

Not applicable. This increment is scaffold-only with no database, ORM, or persistent state.

### API Contracts

Minimal -- only a health endpoint is scaffolded:

- `GET /health` -- Returns `{ "status": "ok" }` with HTTP 200. Used by Docker health checks and smoke tests.
- `GET /api/docs` -- Swagger UI rendering the OpenAPI spec (auto-generated by NestJS/Swagger decorators).

No other endpoints are in scope.

### Architecture Decisions

**AD-01: Turborepo over Nx**
Turborepo provides zero-config monorepo orchestration with remote caching. For a scaffold with simple dependency graphs, Turborepo's lightweight approach avoids Nx's heavier abstraction layer. Trade-off: Nx has richer code generation, but that is not needed at scaffold stage.

**AD-02: pnpm over npm/yarn**
pnpm provides strict dependency isolation (no phantom dependencies), disk-efficient storage via content-addressable store, and native workspace support. This is a hard requirement from the spec.

**AD-03: App Router over Pages Router**
Next.js 15 defaults to App Router. Using it aligns with the framework's direction and enables React Server Components, Suspense boundaries, and streaming SSR when business logic is added later.

**AD-04: Separate `packages/shared` and `packages/ui`**
`packages/shared` contains pure TypeScript (types, utils) with no React dependency -- importable by the NestJS API. `packages/ui` contains React components and depends on `packages/shared`. This prevents the API from pulling in React as a transitive dependency.

**AD-05: Tooling configs as workspace packages**
ESLint, TypeScript, and Prettier configs live in `tooling/*` as internal packages. Apps and packages reference them via workspace protocol (`workspace:*`). This avoids config duplication and ensures a single source of truth for lint rules, TS compiler options, and formatting.

**AD-06: tsup for packages/ui build**
`packages/ui` needs to produce both ESM and CJS output for broad consumption. tsup (powered by esbuild) provides fast, zero-config bundling for TypeScript libraries. Alternative: unbuild -- similar capability but tsup has better React/JSX support out of the box.

**AD-07: Docker Compose for local dev only**
Docker Compose orchestrates all three services with volume mounts for hot reload. No production-optimized images are in scope. Each service runs its dev server inside the container, with source code mounted from the host.

## Implementation Phases

### Phase 1: Monorepo Foundation (US-001, US-006)

1. Initialize pnpm workspace with `pnpm-workspace.yaml` declaring `apps/*`, `packages/*`, `tooling/*`
2. Create root `package.json` with Turborepo as devDependency
3. Create `turbo.json` with `build`, `lint`, `typecheck`, `test`, and `dev` pipelines
4. Create tooling config packages:
   - `tooling/typescript/` -- base `tsconfig.json` with strict mode, path aliases, ESM target
   - `tooling/eslint/` -- ESLint v9 flat config with TypeScript plugin, import plugin
   - `tooling/prettier/` -- Prettier config (trailing commas, single quotes, semicolons)
5. Root `.prettierrc` and `.eslintrc` that reference tooling packages

### Phase 2: Shared Packages (US-005)

1. `packages/shared/` -- TypeScript library exporting placeholder types (`AppConfig`, `ApiResponse<T>`) and a utility function (`formatCurrency`)
2. `packages/ui/` -- React component library with tsup build, exporting a placeholder `Button` component
3. Both packages extend the shared tsconfig base
4. Verify cross-package imports compile

### Phase 3: Applications (US-002, US-003, US-004)

1. `apps/web/` -- Next.js 15 App Router with:
   - Root layout, placeholder home page
   - PWA manifest (`public/manifest.json`) with ShopVui metadata
   - Service worker registration entry point (`next.config.ts` with PWA plugin or manual registration)
   - Imports `Button` from `packages/ui` on the home page
   - Port: 3000

2. `apps/admin/` -- Next.js 15 App Router with:
   - Root layout, placeholder dashboard page
   - Imports `Button` from `packages/ui`
   - Port: 3001

3. `apps/api/` -- NestJS with:
   - Health module (`GET /health` returns `{ status: "ok" }`)
   - Swagger setup at `/api/docs`
   - Imports types from `packages/shared`
   - Port: 4000

### Phase 4: Docker and Verification (US-007)

1. `docker-compose.yml` with services: `web`, `admin`, `api`
2. Volume mounts for source code (hot reload)
3. Port mappings: 3000 (web), 3001 (admin), 4000 (api)
4. Verify `docker compose up` starts all services
5. Verify `pnpm turbo build`, `pnpm turbo lint`, `pnpm turbo typecheck` all pass

## Turbo Pipeline Configuration

```jsonc
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": ["**/.env.*local"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "lint": {
      "dependsOn": ["^build"]
    },
    "typecheck": {
      "dependsOn": ["^build"]
    },
    "test": {
      "dependsOn": ["^build"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

Key: `^build` means "build my dependencies first." This ensures `packages/shared` and `packages/ui` are built before apps that consume them.

## Testing Strategy

Since this is scaffold-only, testing is minimal but must prove the structure works:

- **Unit tests (Vitest)**: Each package and app includes at least one test file confirming exports work. The NestJS health endpoint gets a controller test.
- **Build verification**: `pnpm turbo build` with zero errors is the primary quality gate.
- **Lint/typecheck verification**: `pnpm turbo lint` and `pnpm turbo typecheck` must pass with zero errors.
- **Docker smoke test**: `docker compose up` must bring all services to a reachable state.

Test stack: Vitest for all workspaces (unit). Playwright is not applicable for this increment (no user-facing features to E2E test beyond build verification).

Coverage target: 80% as defined in metadata, though meaningful coverage is limited in a scaffold increment.

## Technical Challenges

### Challenge 1: pnpm workspace protocol with NestJS

NestJS uses its own module system which can conflict with pnpm's strict hoisting. The `packages/shared` import in `apps/api` must work with NestJS's module resolution.

**Solution**: Configure `packages/shared` with proper `exports` field in `package.json` and ensure `tsconfig.json` paths align. NestJS's `paths` in `tsconfig.json` should reference the workspace package.

**Risk**: Low. Well-documented pattern in Turborepo examples.

### Challenge 2: PWA setup in Next.js 15 App Router

The `next-pwa` package has historically lagged behind Next.js releases. Next.js 15 App Router may require manual service worker registration.

**Solution**: Use `@ducanh2912/next-pwa` (actively maintained fork) or manual service worker registration via `next.config.ts`. Start with a static `manifest.json` in `public/` and a minimal service worker entry point. Full offline caching is out of scope.

**Risk**: Medium. If no PWA plugin supports Next.js 15 at time of implementation, fall back to manual registration.

### Challenge 3: Docker volume mounts + pnpm node_modules

pnpm's content-addressable store does not play well with naive Docker volume mounts. Mounting host `node_modules` into a container can cause platform-specific binary mismatches.

**Solution**: Use a multi-stage approach: install dependencies inside the container (not mounted from host), and only mount source code directories. Use `.dockerignore` to exclude `node_modules`. The `docker-compose.yml` mounts specific source directories, not the entire monorepo root.

**Risk**: Low. Standard pattern for pnpm + Docker.

## Domain Skill Recommendations

This increment is infrastructure scaffolding with no business logic. Domain skill complexity is minimal:

- **No frontend:architect needed** -- the Next.js apps are placeholder pages with no state management, routing logic, or component architecture to design.
- **No backend:* needed** -- the NestJS app is a health endpoint with Swagger. No service layers, repositories, or domain patterns.
- **Implementation** can proceed directly with `/sw:do` after tasks are generated via `/sw:test-aware-planner`.

When future increments add business logic (product catalog, auth, cart), those will warrant `frontend:architect` and `backend:nodejs` skill invocations.
